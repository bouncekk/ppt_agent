技术选型：   
| 模块     | 选型                         | 说明       |
| ------ | -------------------------- | -------- |
| 后端框架   | FastAPI                    | 异步 + 服务化 |
| LLM    | DeepSeek 云 API             | 可替换      |

阶段开发计划表（示例）

| 阶段 | 时间范围 | 主要任务 | 关键产出 | 负责人 |
| --- | --- | --- | --- | --- |
| 阶段一：需求梳理与总体架构设计 | 第 1–3 天 | - 明确用户使用流程（上传 PPT → 解析 → 扩展 → 查看/搜索） - 选定技术栈：后端框架（FastAPI）、LLM 接入方式、向量库（Milvus/Pinecone/Chroma 等）、PPT 解析库（Unstructured/PyMuPDF + python-pptx 等） - 画出系统架构图：包含前端/后端、向量库、外部搜索接口、Agent 工具链、Checklayer | - 《架构设计初稿》（架构图+组件说明） - 项目仓库初始化（README 草稿） | 架构/工程组 |
| 阶段二：文档解析与数据建模 | 第 4–7 天 | - 实现 PPT 文件上传/读取（本地文件） - 使用 python-pptx解析 PPT 文本结构（目录、主标题、子标题、正文、图片说明） - 设计内部数据结构：如 Slide{index,title,bullets,notes} - 将解析结果持久化（JSON / DB） | - 可运行的“PPT → 结构化 JSON”模块 - 单元测试或脚本验证若干 PPT 样例 | 算法/Agent 组 + 工程组 |
| 阶段三：向量化存储与语义检索 | 第 8–11 天 | - 选定并搭建向量数据库（Milvus / Pinecone / 本地向量库） - 对每一页/每个知识点生成向量（调用嵌入模型） - 实现“给定某页/某标题 → 找到相关内容”的检索接口 - 初步 REST API：/upload, /slides, /search | - 向量库服务跑通（本地 or 云端） - 语义检索 API 可用 | 算法/Agent 组 |
| 阶段四：LLM 扩展与 Agent 工具链设计 | 第 12–16 天 | - 设计 Agent 的工具链： - 工具1：根据 slide 内容与检索结果，写扩展讲解/公式推导/代码示例 - 工具2：多源搜索（Wikipedia / Arxiv / 学术 API）封装 - 工具3：Checklayer（先做 self-consistency再做校验事实、过滤离谱内容） - 设计并固化 Prompt 模板（包含角色设定、输出格式要求、语言风格） - 联调：输入 PPT 后，可以为每一页生成扩展笔记 | - Agent 设计文档（工具列表+Prompt 模板） - 核心扩展功能可跑通（CLI 或简单接口） | 算法/Agent 组 |
| 阶段五：前端/交互与整体联调 | 第 17–20 天 | - 实现一个简单的 Web 界面或 CLI：上传 PPT、查看每页扩展内容、点击某一页查看“多维搜索结果” - 集成：上传 → 解析 → 向量化 → 检索 → LLM 扩展 → 展示 - 加入基础容错（上传失败、模型超时、检索为空等） | - Demo 级完整流程 - UI 原型/简单页面 | 工程组 |
| 阶段六：云原生 & 工程化完善 | 第 21–24 天 | - 编写 Dockerfile，保证在标准环境可一键构建 + 运行 - 根据时间选择： - 本地 Docker Compose 或 K8S 简要部署说明 - 使用 Redis/消息队列 等简要展示“云原生特性”（可选） - 完善 README：启动步骤、环境变量说明、依赖说明 | - 可构建并运行的 Docker 镜像 - 完整 README & 启动指南 | 架构/工程组 |
| 阶段七：技术文档 & 演示视频 | 第 25–28 天 | - 整理技术文档： - 架构设计（架构图、组件、数据流） - LLMAgent 策略与 Prompt 设计过程 - 云原生特性说明 - 分工说明（每人百分比 + 模块） - 录制 3–5 分钟演示视频：用一个真实 PPT 展示完整流程，并讲 1 分钟架构 | - 《技术文档》PDF/MD - 演示视频（含架构讲解） - 分工文档 | 全体 |
